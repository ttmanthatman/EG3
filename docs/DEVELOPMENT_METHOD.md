# Development Method

本文档定义本项目的固定开发方法。除非 Maxiao 明确确认修改，否则后续开发按本文档执行。

## 角色分工

- **Maxiao（架构师）**：负责架构意图、业务判断、验收标准和关键取舍。不需要写代码。
- **Flow / 心流（AI 技术搭档）**：负责把架构意图落到文档、代码、测试、部署和版本管理。当信息不足时必须提问，不能猜测用户真正想做什么。

## 每次开始前的固定步骤

1. 读取 `README.md`、`docs/DEVELOPMENT_METHOD.md`、`docs/AI_NAMING_REGISTRY.md`、`docs/SYSTEM_FLOW.md`。
2. 检查 Git 状态，确认是否有未提交变更。
3. 明确本轮目标，只处理本轮需要的模块，不一次性展开所有模块。
4. 如果涉及新架构或新技术，动手前调研是否已有类似项目、可复用方案、常见坑和经验。
5. 如果架构意图不清楚，先问 Maxiao。

## 前置文档规则

### 文档 1：AI 命名登记表

所有命名都进入 `docs/AI_NAMING_REGISTRY.md`，包括：

- 模块名、文件名、变量名、函数名、类名
- 数据字段名、API 路由名、数据库表名、外部服务名

登记目的不是给用户阅读，而是让 AI 在长期开发中保持命名一致。

### 文档 2：系统流程文档

所有调用关系、数据流和模块协作进入 `docs/SYSTEM_FLOW.md`。该文档必须让 Maxiao 能看懂，同时也让 AI 能继续开发。

每次涉及架构变化时，必须同步更新 Mermaid 图。

## 调研规则

当 Maxiao 确认具体架构后，动手写代码前必须调研：

- 是否已有可复用项目或库
- 类似系统的常见架构
- 数据流和状态管理常见坑
- 部署和运维经验

调研结果写入 `docs/RESEARCH_NOTES.md`（持续追加，不重写）。

## 错误勘验规则

当 Maxiao 指出 AI 的实现、解释或架构理解有错误时，不能只修表面现象。必须先做错误勘验：

1. 复述错误，确认发生在哪个边界、模块或流程
2. 区分三类问题：规则理解错误 / 实现边界错误 / 验证标准错误
3. 找到根因，不是只找触发点
4. 检查同类问题是否残留在代码、文档、UI、测试、命名表和流程图里
5. 修改代码和文档，补充防止同类错误复发的验证项
6. 勘验结果写入 `docs/ERROR_INSPECTIONS.md`
7. 提交 Git 版本

## 版本回溯规则

每一个完成的小步骤都要形成可回溯版本：

1. 文档变更完成后提交一次
2. 代码实现完成后提交一次
3. 测试或部署配置完成后提交一次
4. 部署成功后记录部署说明，再提交一次

提交信息格式：`type: description`，例如 `docs: establish development method`。

## GitHub 同步规则

- 仓库：`https://github.com/ttmanthatman/EG3`
- 每次 commit 后 push 到 GitHub
- 本地和远程必须保持同步，方便 Maxiao 换机器继续工作

## 部署规则

- 部署目标：`3.xiaogushi.us`（VPS: 45.62.123.222）
- 不触碰 VPS 上 `xiaogushi.us` 的其他服务
- 不把服务器密码写进仓库或日志文档
- 部署前先备份即将修改的线上配置
- 部署后记录部署时间、提交版本、服务位置和回滚方式

## Grill-me 使用规则

当出现以下情况时启用 `grill-me`：

- 模块边界不清楚
- 数据所有权、状态流向、调用链不明确
- MVP 范围可能膨胀
- 一个决策会影响长期架构

启用后，AI 需要连续追问，直到模块边界、输入输出和验收标准明确。
